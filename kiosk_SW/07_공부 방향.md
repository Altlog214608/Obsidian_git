2. **React 심화 학습**
   ```typescript
   // 기존 방식
   function App() {
     const [count, setCount] = useState(0);
     return <div onClick={() => setCount(count + 1)}>{count}</div>;
   }

   // 개선된 컴포넌트 구조
   interface CounterProps {
     initialValue: number;
     onCountChange: (newCount: number) => void;
   }

   const Counter: React.FC<CounterProps> = ({ initialValue, onCountChange }) => {
     const [count, setCount] = useState(initialValue);
     
     const handleClick = useCallback(() => {
       const newCount = count + 1;
       setCount(newCount);
       onCountChange(newCount);
     }, [count, onCountChange]);

     return <div onClick={handleClick}>{count}</div>;
   };
   ```

다음 순서로 파일들 열어보세요 — 우선순위·이유·짧은 확인포인트와 VS Code로 여는 명령을 함께 둡니다.

1) Home.tsx — (우선)  
- 이유: App의 "/"가 이 컴포넌트를 마운트하므로 첫화면의 무거운 초기화(3D, 이미지, 트래킹 등)를 여기서 하는지 확인해야 함.  
- 확인포인트: mount 시점의 side-effect(useEffect), heavy 라이브러리(ogl, swiper 등) 초기화, 트래킹 호출 위치, 리소스 경로.  
- 열기:
```powershell
code src\renderer\src\pages\Home.tsx
```

2) session.ts — (App에서 startNewSession 호출 관련)  
- 이유: 세션 생성 로직(중복 방지, id 생성 방식, mixpanel.reset 등)을 확인해야 세션 분리 정책이 안전한지 알 수 있음.  
- 확인포인트: crypto.randomUUID 사용 여부, mixpanel.reset/identify 호출 타이밍, 비동기 처리 안전성.  
- 열기:
```powershell
code src\renderer\src\tracking\session.ts
```

3) index.ts (또는 mixpanel.ts) — (트래킹 전체)  
- 이유: App 시작 시 trackEvent 호출이 트래킹 초기화 전에 발생하면 이벤트 유실 가능. 초기화/전송 흐름 확인.  
- 확인포인트: 초기화 조건(VITE_MIXPANEL_TOKEN), 큐잉/버퍼링, dev/prod 분기, 민감정보 처리.  
- 열기:
```powershell
code src\renderer\src\tracking\mixpanel.ts
code src\renderer\src\tracking\index.ts
```

4) viewport.ts — (뷰포트/스케일링)  
- 이유: AppRoutes가 transform scale을 사용하므로 계산 로직(스크린 사이즈→scale)이 UI 레이아웃/픽셀 정렬에 중요.  
- 확인포인트: 계산 공식, center/transformOrigin, rounding/subpixel 처리, 모바일/큰 화면 동작.  
- 열기:
```powershell
code src\renderer\src\config\viewport.ts
```

5) main.tsx — (렌더러 진입점)  
- 이유: React 앱 마운트, 트래킹 초기화 순서, Provider 래핑 등 전체 초기화 순서를 파악.  
- 확인포인트: 트래킹 init 위치, global CSS/스타일 로드, 서비스 초기화.  
- 열기:
```powershell
code src\renderer\src\main.tsx
```

6) index.ts 및 index.ts — (보안/IPC 관점)  
- 이유: renderer가 호출하는 window.api/window.electron과 main의 IPC 핸들러가 일치하는지, 보안 설정 확인 필요.  
- 확인포인트(preload): 노출 API 목록, 타입 선언 일치.  
- 확인포인트(main): BrowserWindow webPreferences, preload 경로, IPC handle 등록.  
- 열기:
```powershell
code src\preload\index.ts
code src\main\index.ts
```
